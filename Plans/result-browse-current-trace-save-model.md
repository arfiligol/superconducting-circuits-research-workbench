# Result Browse Current-Trace Save Model

This plan records the product-model correction for `Circuit Simulation` result stages.
It does not replace `Plans/local-mode-research-usability-sprint.md`; it refines the next execution slice.

## 0) Task Information

- Agent: `Planning Agent`
- Task ID / Topic: `L10-Result-Browse-And-Current-Trace-Save`
- Status: `Active planning checkpoint 2026-03-20`

## 1) Goal

Make `Simulation Result` and `Post Processing Result` behave like result browsers, not task-level bulk publication surfaces.

The target researcher flow is:

1. browse the current result bundle
2. choose the exact trace they want to inspect
3. save that current trace into a design
4. reopen the saved trace in `Raw Data`

## 2) Product Decisions

### A. `Post Processing Setup` must not own `Raw / PTC`

- Stage 4 should define how post-processing runs, not how results are browsed afterward.
- Remove `Raw / PTC`, family, metric, and port browsing controls from `Post Processing Setup`.
- Keep Stage 4 focused on:
  - steps
  - step order
  - optional note

### B. `Post Processing Result` must own source and family browsing

- Stage 5 should be the place where the researcher switches between available processed-result views.
- Result browsing should allow:
  - `Raw` -> `S / Y / Z`
  - `PTC` -> `Y / Z`
- `PTC` availability must still be gated by persisted upstream capability.

### C. `Simulation Result` and `Post Processing Result` should save the current selected trace, not the whole task bundle

- Current save behavior is too coarse.
- Saving should not publish every trace generated by the task.
- Saving should target the current selected trace shown in the explorer.
- The saved object should be the canonical trace identity, not the current metric-only visualization.

### D. Metric is a view control, not the saved object identity

- `Real / Imaginary / Magnitude / dB / Phase` are explorer views.
- They should not create separate saved trace records by themselves.
- Current-trace save should be anchored to stable trace identity such as:
  - source
  - family
  - output/input port
  - mode group / mode if relevant

### E. Save interaction should move into the explorer surface

- `Save to Design` should no longer be a separate large support card under `Simulation Result`.
- Prefer a compact action near the result explorer controls.
- Recommended direction:
  - place `Save Current Trace` near the family row / explorer controls
  - click opens a compact dialog
  - dialog handles:
    - existing design selection
    - `New Design`

### F. Active Dataset remains shell-owned context

- The save dialog must not reintroduce dataset-target editing.
- Dataset context should stay implicit unless a conflict or recovery condition forces it visible.

## 3) Current Implementation Mismatch

### Simulation Result

- Current save path is still task-level publication.
- Backend publication materializer currently emits a synthetic bundle per task, not a user-selected current trace.
- Current frontend `SimulationResultPublicationCard` assumes bundle publication and therefore no longer matches the intended model.

### Post Processing Setup / Result

- Current Stage 4 still persists `source` inside `post_processing_setup`.
- Current Stage 5 explorer behavior is therefore partially locked by setup-level source.
- This is now considered the wrong owner boundary.

## 4) Backend Work Required

### 4.1 Add a trace-scoped publication contract

- Replace or supplement the current task-level simulation publication API.
- The new publication path must accept the current result selection.
- Required direction:
  - target task id
  - target design id
  - current trace selection payload

Selection payload should be based on stable explorer identity, not only the rendered metric.

### 4.2 Give explorer responses a stable publishable trace identity

- Explorer payload must return enough metadata for frontend save to be truthful.
- The backend must know which saved trace the current explorer selection refers to.
- Current synthetic publication materializer is too coarse because it only publishes one row per family/source bundle.

### 4.3 Remove `source` from `post_processing_setup`

- `post_processing_setup` should no longer persist result-browse source.
- Retry/recovery of Stage 4 should restore steps and note only.

### 4.4 Ensure post-processing result browse can expose both `Raw` and `PTC`

- The result surface must expose all persisted result sources available for the processed bundle.
- Family availability must follow source truthfully:
  - `Raw`: `S / Y / Z`
  - `PTC`: `Y / Z`

## 5) Frontend Work Required

### 5.1 Remove browse controls from Stage 4

- Delete source/family/metric/port controls from `Post Processing Setup`.
- Keep the stage visually about building steps.

### 5.2 Make Stage 5 the authoritative result browser

- `Post Processing Result` should host:
  - source
  - family
  - metric
  - port controls
  - plot/table view

### 5.3 Replace the large simulation save card

- Remove or retire the current `SimulationResultPublicationCard`.
- Replace it with compact explorer-local save affordance:
  - `Save Current Trace`
  - small dialog for design targeting

### 5.4 Reuse the same save interaction in both result stages

- `Simulation Result` and `Post Processing Result` should use one shared selection-aware save model.
- Prefer shared primitives over separate duplicated cards.

## 6) Implementation Slices

### L10-A Backend trace-scoped publish contract

- Add publish-by-selection backend API
- add stable explorer trace identity
- update persistence/materialization for single-trace save

### L10-B Frontend explorer-integrated save

- remove large `Save to Design` card
- add compact `Save Current Trace` action + dialog
- wire simulation result explorer to current-trace save

### L10-C Post-processing owner-boundary correction

- remove source browsing from Stage 4
- move source/family browsing fully into Stage 5
- support `Raw` and `PTC` in Stage 5 truthfully

### L10-D Live verification

- simulation result current-trace save
- raw-data reopen on saved trace
- post-processing result current-trace save
- refresh / reattach continuity

## 7) Open Technical Questions To Settle During Implementation

1. What exact backend request shape should represent `current trace`?
2. Is trace identity based on family/source/output_port/input_port only, or are there additional mode selectors that must be part of the key?
3. Should save open a shared dialog component used by both result stages?
4. For `Post Processing Result`, does the local execution payload already materialize enough source/family variants, or must backend synthetic result generation be widened first?
