# Shared App Model

This area includes the App-level shared model that Frontend and Backend both depend on.

<Aside type="note" title="What belongs here">

If a document is also responding to shell context, workspace collaboration, task execution visibility, runtime governance, or audit trail,
But it does not belong to a single frontend page or a single backend surface, so it should be placed here.

</Aside>

<Aside type="caution" title="Not Core, Not CLI">

This zone is neither `Core` nor `CLI`.
These pages mainly define the shared semantics of multi-user apps and service-backed workflows.

</Aside>

<Aside type="note" title="Same app across local and online modes">

This area is also responsible for the mode model shared between `Local Mode` and `Online Mode` for the same App.
If a contract needs to answer "whether local bypass auth" or "whether online requires collaboration", it usually belongs here.

</Aside>

## Grouping Principle

This area gives priority to grouping by **responsibility type** instead of splitting trees according to `Local Mode` / `Online Mode`.

The reason is:

- Most shared contracts apply to both `local` and `online`
- The difference is usually applicability or authority semantics, not that the owner has really changed his mind
- If you directly split it into a mode-based folder / nav, it will be easy to have two duplicate SoTs.

<Aside type="note" title="Mode is an applicability dimension">

`Local Mode` / `Online Mode` should give priority to expressing applicability and difference semantics in the contract on each page.
Shared index and nav give priority to answering "What responsibilities does this contract manage?"

</Aside>
