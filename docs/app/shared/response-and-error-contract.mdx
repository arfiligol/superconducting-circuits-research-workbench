# Response & Error Contract

This page defines the response envelope and error code contract shared by App backend surfaces. The frontend should use this page as a common baseline for displaying and triaging errors.

<Aside type="note" title="Shared backend contract">

`Session & Workspace`, `Tasks & Execution`, `Circuit Definitions`, `Analysis Results` should all return to this shared envelope.

</Aside>

<Aside type="caution" title="Frontend keys off codes, not messages">

Front-end UI should not rely on message text comparison to determine display behavior.
The official split keys are `error.code`, `error.category` and `error.retryable`.

</Aside>

## Success Envelope

| Field | Required | Meaning |
|---|---|---|
| `ok` | required | Fixed to `true` when success |
| `data` | required | surface-specific payload |
| `meta` | optional |collection pagination, generated_at, request echo, context summary|

<Aside type="note" title="Success envelope">

```json
{
 "ok": true,
 "data": {
  "workspace": {
   "id": "ws_lab_a",
   "name": "Lab A"
  }
 },
 "meta": {
  "generated_at": "2026-03-14T10:00:00Z"
 }
}
```

</Aside>

## Collection Meta Baseline

In response to list / catalog / history / task execution classes, `meta` should prefer cursor-based collection meta.

| Field | Meaning |
|---|---|
| `generated_at` | response generation time |
| `limit` | The target number of pens for this response |
| `next_cursor` | Next page cursor; if there is no more information, it can be `null` |
| `prev_cursor` | Look back at the cursor of the previous data; if it is not supported or is already at the starting point, it can be `null` |
| `has_more` | Is there still more information |
| `filter_echo` | backend actual filter / scope / sort |

<Aside type="tip" title="Cursor-based by default">

The dynamic list of App backend should prefer cursor-based pagination instead of `page/page_size`.
This especially applies to task execution history, run history, audit logs, and catalog / browse surfaces that change continuously.

</Aside>

## Error Envelope

| Field | Required | Meaning |
|---|---|---|
| `ok` | required | fixed to `false` when error occurs |
| `error.code` | required |stable machine-readable error code|
| `error.category` | required | Shared high-level classification |
| `error.message` | required | user-safe message |
| `error.retryable` | required | Is it safe to retry |
| `error.details` | optional | redaction-safe structured detail |
| `error.debug_ref` | optional |correlation id, task id or request id|

<Aside type="note" title="Error envelope">

```json
{
 "ok": false,
 "error": {
  "code": "dataset_not_visible_in_workspace",
  "category": "permission_denied",
  "message": "The selected dataset is not visible in the active workspace.",
  "retryable": false,
  "details": {
   "dataset_id": "ds_123",
   "workspace_id": "ws_lab_a"
  },
  "debug_ref": "req_7b4c4e"
 }
}
```

</Aside>

## Common Error Categories

This page follows the high-level classification of [Code Quality / Error Handling](../../reference/guardrails/code-quality/error-handling.mdx).

| Category | Meaning |
|---|---|
| `auth_required` | Not logged in or session is unavailable |
| `permission_denied` | Logged in, but current identity / workspace does not have permission to execute |
| `validation_error` | The request payload or surface-specific input is illegal |
| `not_found` | The target resource does not exist or cannot be resolved |
| `conflict` | Version conflict, status conflict, ownership conflict |
| `task_not_ready` | The result or control action is not ready yet |
| `task_execution_failed` | runtime/execution itself failed |
| `persistence_error` | storage/persistence write or read failed |
| `internal_error` | Uncategorized server error |

## Shared Error Code Families

| Family | Representative codes |
|---|---|
| Session / Workspace | `workspace_membership_required`, `workspace_invite_expired`, `workspace_invite_revoked`, `workspace_invite_account_mismatch`, `active_workspace_required`, `dataset_not_visible_in_workspace`, `context_rebind_required` |
| Tasks | `task_not_found`, `task_not_visible`, `task_submit_denied`, `task_not_cancellable`, `task_not_terminable`, `task_already_terminal`, `task_retry_denied` |
| Definitions | `definition_not_found`, `definition_not_visible`, `definition_source_invalid`, `definition_conflict`, `definition_delete_blocked` |
| Analysis Workbench | `analysis_not_available`, `trace_selection_invalid`, `artifact_not_found`, `tagging_conflict` |

## HTTP Status Mapping Baseline

| Situation | HTTP status |
|---|---|
| auth required | `401` |
| permission denied | `403` |
| validation error | `400` |
| not found | `404` |
| conflict | `409` |
| task not ready | `409` |
| persistence error | `503` or `500` |
| internal error | `500` |

## Frontend Display Rules

| Signal | Frontend expectation |
|---|---|
| `error.code` | Driver page-specific UI state and action disable / retry affordance |
| `error.retryable = true` | Can display retry CTA or polling guidance |
| `error.debug_ref` | Not directly used as user message, but can be associated with support / audit / logs |
| `error.details` | Only used for safe structured rendering, do not display raw internals |

<Aside type="tip" title="Surface-specific pages still define their own codes">

This document only defines envelopes and shared families.
Each backend surface still needs to list its own concrete error codes and example payload.

</Aside>
