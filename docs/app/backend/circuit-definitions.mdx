# Circuit Definitions

This page defines the backend circuit definition surface that frontend `Schemas` / `Schema Editor` depends on.

<Aside type="note" title="Surface Boundary">

This page only defines definition catalog, detail, mutation and persisted inspection read model.
Simulation and analysis run are not the responsibility of this page.

</Aside>

<Aside type="tip" title="Primary Consumers">

The main consumers are [Schemas](../frontend/definition/schemas.mdx), [Schema Editor](../frontend/definition/schema-editor.mdx) and [Circuit Simulation](../frontend/simulation-workbench/circuit-simulation.md).

</Aside>

<Aside type="caution" title="UUIDv4-only Schema Identity">

- `definition_id` is persisted schema identity and must be full UUIDv4
- identity is opaque and no establishment order, stable display order or human ranking can be inferred from the value itself
- If the UI requires readability, it can display short `Schema ID`, but the canonical persisted identity is still full UUIDv4
- The identification of schema with the same name must be based on short `Schema ID`, `created_at`, and owner / workspace context if necessary

</Aside>

<Aside type="caution" title="UUIDv4-only contract">

- Currently SoT does not recognize numeric definition identity
- runtime path must not preserve dual-ID compatibility
- catalog, editor, simulation and task binding all operate with UUIDv4-only `definition_id`

</Aside>

<Aside type="caution" title="Database version control uses Alembic">

- backend metadata DB schema versioning managed by Alembic
- `bootstrap_metadata_schema()` Execute `alembic upgrade head` in app startup
- `alembic_version` is the schema-version authority of metadata DB
- reset / rebuild tools are not a canonical schema-version strategy

</Aside>

## Persistence And Schema Version Contract

| Concern | Contract |
|---|---|
| Metadata DB schema versioning | Alembic revision chain is authoritative |
| App startup bootstrap | `bootstrap_metadata_schema()` runs Alembic upgrade to `head` before repositories/services consume the DB |
| Persisted identity changes | must ship as Alembic revisions recorded in the metadata DB schema-version chain |
| Numeric `definition_id` payloads | rejected by the API contract; UUIDv4 is the only valid persisted identity shape |
| Reset / rebuild tools | development-only recovery path; not the canonical production or team schema-version story |

## Surface Contracts

<Tabs>
<TabItem label="Catalog">


The catalog list must support at least the following functions:

| Function | Description |
|:--- |:--- |
| **Search** | Keyword search |
| **SORT** | Sort by date, name, or other attributes |
| **Browse** | Load large amounts of data in sections using cursor-based methods |
| **Metadata** | Contains only summary-safe metadata |

catalog request / response baseline:

| Field | Meaning |
|---|---|
| `workspace_id` | Determined by active session context; local mode is fixed to `Local Space` and cannot be specified by frontend across workspaces |
| `search_query` | optional |
| `sort_by` / `sort_order` | optional |
| `limit` | optional |
| `after` / `before` | Optional cursor-based browse position |
| `rows[]` | `definition_id`, `name`, `created_at`, optional `updated_at`, `visibility_scope`, `owner_display_name`, `allowed_actions` |
| `meta.next_cursor` / `meta.prev_cursor` | cursor-based browse meta |
| `total_count` | Total Summary |

</TabItem>

<TabItem label="Detail">


A single definition detail must provide at least:

| Categories | Included Items |
|:--- |:--- |
| **Source** | canonical source text |
| **Identity** | schema name |
| **Validation** | validation notices, validation summary |
| **Result** | normalized output, preview artifacts summary |

The detail response must also contain at least:

| Field | Meaning |
|---|---|
| `definition_id` |persisted schema identity; full UUIDv4|
| `workspace_id` / `visibility_scope` | shared boundary visibility; local mode can be fixed to `Local Space` / `local` |
| `allowed_actions` |`update`, `delete`, `publish`, `clone` and similar values|
| `created_at` |identity disambiguation and stable ordering metadata|
| `updated_at` | concurrency / freshness summary |

</TabItem>

<TabItem label="Mutation">


The backend must support standard CRUD operations:

1. **Create**: Create a new definition
2. **Update**: Update existing definition
3. **Delete**: Delete existing definition

mutation payload baseline:

| Mutation | Required fields |
|---|---|
| `create` | `name`, `source_text`, optional `visibility_scope`; local mode defaults to `local` |
| `update` | `definition_id`, `source_text`, optional `name`, optional concurrency token |
| `delete` | `definition_id` |
| `publish_to_workspace` | `definition_id` |
| `clone_private_copy` | `definition_id`, optional `name` |

</TabItem>

</Tabs>

## Identity And Ordering Contract

| Concern | Contract |
|---|---|
| `definition_id` | full UUIDv4 persisted identity; opaque, unorderable inference |
| `Schema ID` display | UI convenience only; UUID prefix is ​​desirable, but may not replace full `definition_id` |
| Human ordering | Must use fields such as `created_at`, `updated_at`, `name`; cannot use `definition_id` value sorting |
| Same-name disambiguation | Display at least short `Schema ID` + `created_at`; add owner / workspace context if necessary |
| Selection / mutation binding | catalog, editor, schemdraw, simulation, task submit are all bound with full `definition_id` |

## Workspace And Visibility Rules

| Rule | Meaning |
|---|---|
| One workspace per definition | persisted definition belongs to only one workspace |
| Default create scope | local mode is defaulted to `Local Space` and `local`; online mode is defaulted to current active workspace and `private` |
| Visibility filtering | catalog / detail returns only definitions visible to the current session |
| Publish is explicit | `private -> workspace` is only available in online mode; not available in local mode publish / share |

<Aside type="caution" title="Catalog is Summary-only">

definition catalog MUST NOT treat the full definition payload as part of the list response.
The list path only provides `id`, `name`, `created_at` and equivalent summary fields.

</Aside>

<Aside type="caution" title="Do not use schema identity as display order">

The UUIDv4 value of `definition_id` has no human ordering meaning.
If the UI needs to display the "latest" or "older" schema, it must be calculated based on time fields such as `created_at` / `updated_at`.

</Aside>

<Aside type="caution" title="Persisted Preview Authority">

`validation notices`, `normalized output` and `preview artifacts` only represent the persisted state after the last successful save.
backend must not treat an unsaved editor draft as a formal preview authority.

</Aside>

<Aside type="tip" title="Persistence Sync">

After the mutation is successful, when the frontend re-reads the detail, it must be able to see the latest persisted inspection result.

</Aside>

<Aside type="caution" title="Destructive Mutation">

Deleting a definition is a **destructive mutation**.
The backend should maintain explicit success/failure responses and should not be replaced by silent ignores.

</Aside>

## Delivery Rules

| Project | Rules |
|:--- |:--- |
| **Source of Truth** | canonical source text is the only writing authority. |
| **Derived Preview** | normalized output and preview artifacts are read models and cannot be written back to the source. |
| **Mutation Refresh** | After create / update / delete, both catalog and detail must be able to read back consistent results. |
| **Reuse** | The same definition detail must be shared between schema editor and circuit simulation. |

## Error Code Contract

| Code | Category | When it applies |
|---|---|---|
| `definition_not_found` | `not_found` | Target definition does not exist |
| `definition_not_visible` | `permission_denied` | definition is not visible for the current session |
| `definition_source_invalid` | `validation_error` | source text failed validation |
| `definition_conflict` | `conflict` | concurrency token or persisted version conflict |
| `definition_delete_blocked` | `conflict` | definition cannot be deleted at this time |
