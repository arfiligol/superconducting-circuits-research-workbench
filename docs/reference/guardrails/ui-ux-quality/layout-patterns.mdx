# Layout Patterns

<Aside type="note" title="Use this page for shell and page-structure decisions">

This page answers where the layout boundary should be placed, and how the data-dense pages should be arranged. It does not replace page-level spec.

</Aside>

## Layout Map

| Hierarchy | Key Responsibilities |
| --- | --- |
| Root layout |providers, theme, fonts, global styles|
| Workspace layout |sidebar, top bar, shared workspace context|
| Feature layout |tabs, breadcrumb, sub-navigation|

## App Router Responsibilities

- Root layout:providers, theme, fonts, global styles
- Workspace layout:sidebar, top bar, shared workspace context
- Feature layout:tabs, breadcrumb, sub-navigation

## Route Groups

- `(workspace)`: Main product area
- `(docs)` or other non-main product areas can be grouped independently
- Don’t pile all pages under the root layout

## Data-Dense View Pattern

For data-intensive pages, master-detail is preferred:

- Left side:table / list / search / filters
- Right side: detail panel / chart / analysis output
- Stackable for mobile

<Aside type="tip" title="Good default">

If the page has list, search, filtering and detail/preview at the same time, it is better to start from master-detail instead of stacking everything in a single column first.

</Aside>

## Spacing

- Use consistent spacing scale
- Use medium spacing for page-level blocks
- Card spacing remains tight but readable
- Avoid sacrificing data density for the sake of "looking great"

## Guidance Density

| Prefer | Avoid |
|---|---|
| Guided through hierarchy, grouping, alignment and spacing | Pile a large number of explanatory paragraphs first and then hope users can organize them themselves |
| concise labels, status badges, section titles | Long section helper copy takes up shell or card |
| clear primary action + quiet secondary actions | The same level is filled with a lot of copywriting and CTA |

<Aside type="tip" title="Let layout teach first">

Don't add a paragraph of auxiliary text when things can be explained clearly using layout, partitions, button hierarchy and status positions.
Description text should be supplementary and should not be the main navigation mechanism for the entire UX.

</Aside>

## Single-primary-task Rule

| Rule | Meaning |
|---|---|
| One page, one primary job | Each page body should clearly serve a primary task, such as dataset management, raw-data browse, simulation workflow, schemdraw authoring |
| Secondary surfaces stay secondary | Support information may exist, but must not dilute the main mission or steal attention from the first screen |
| Shell context stays in shell | `Runtime Mode`, `Active Workspace`, `Active Dataset`, `Tasks Queue`, worker summary belong to shared shell and should not be redone on each page context summary wall |
| Cross-page CTA is not IA | `Open X`, `Go to Y`, `Handoff to Z` is not information architecture; should only appear if it is a single major next step |

<Aside type="caution" title="Do not overbuild page bodies">

Don’t shove a piece of information into the page body just because it “looks complete.”
Duplicated shell context, authority summary, handoff cards, navigation button walls, and large explanation copy should not exist if they are not indispensable for completing the main task of this page.

</Aside>

## Noise Budget

| Keep | Remove or demote |
|---|---|
| Clear section hierarchy, necessary state, single primary CTA | Repeated shell context such as runtime mode cards, target dataset cards, submit authority cards, etc. |
| The concise blocking reason required for the main process | A large piece of supplementary text just to "help explain" |
| Results/status directly related to the main task of this page | Cross-page navigation wall, handoff wall, summary cards not directly related to this page |
| Metadata and status that can be quickly scanned | Additional information that turns the page into an admin wall or diagnostics wall |

## Agent Rule
```markdown
## Layout Patterns
- Use App Router layouts intentionally:
  - root layout for providers/theme/fonts
  - workspace layout for shared shell
  - feature layout for sub-navigation
- Use route groups to separate workspace surfaces from other sections.
- Data-dense pages should prefer a master-detail structure with mobile-safe stacking.
- Keep spacing consistent and compact enough for dense data workflows.
- Do not collapse the entire product into one flat page tree without layout boundaries.
- Prefer guidance through layout hierarchy before adding explanatory copy.
- Keep shell and dashboard surfaces low-noise; helper text should be concise and only where ambiguity or risk remains.
- Each page body should serve one primary task; do not dilute it with duplicated shell context or cross-page CTA walls.
- Keep runtime mode, active workspace, active dataset, task queue, and worker summary in shared shell surfaces unless the page cannot function without rendering a task-local subset.
- Use follow-up navigation only when it is the single primary next action, not as a substitute for clear IA.
- Remove or demote authority summaries, handoff cards, and helper panels that are not required to complete the page's primary job.
```
