# Theming

主題系統必須為 Next.js frontend 服務，而不是把顏色散落在 component 內。

<Aside type="note" title="Use this page for theme decisions">

這頁定的是 theme ownership 與 token baseline。若只是單一元件的視覺調整，仍應回到 semantic token，而不是直接硬編碼色值。

</Aside>

## Theme Map

| concern | baseline |
| --- | --- |
| theme runtime | `next-themes` |
| supported modes | `light` / `dark` / `system` |
| color semantics | 用 token，不用 raw utility 或 hex 當產品語意 |
| state preservation | theme switch 不得清空表單或選取狀態 |

## Theme Management

- 使用 `next-themes`
- root layout 提供 `ThemeProvider`
- 提供 light / dark / system 三種模式

## Semantic Tokens

樣式應優先使用語意化 token，例如：

- `background`
- `foreground`
- `card`
- `muted`
- `border`
- `primary`
- `destructive`

禁止直接把 `bg-white`、`text-black`、硬編碼 hex 當成產品語意。

<Aside type="caution" title="Semantic tokens first">

Tailwind utility 可以表達佈局與狀態，但產品色彩語意必須先經過 token。不要讓 component 自己決定整個產品的色彩語言。

</Aside>

## Rules

- Tailwind 可用於佈局與語意 class，不要用來散布硬編碼色彩決策
- 新元件必須在 light / dark 下都可讀
- 圖表主題需跟隨目前 theme
- 任何 theme 切換不可造成表單或選取狀態重置

## Light Mode Direction

| Concern | Baseline |
|---|---|
| Base surface | 以乾淨 white base 或較暖的 neutral base 為主，不應被過冷的 blue-gray wash 蓋住整個 app |
| Accent language | 保留淡 accent fills、soft icon accents 與 restrained semantic states |
| Mood | light mode 應是 dark mode 的 sibling，不是另一個產品語言 |
| Rejection | 不接受大面積偏冷、偏藍、像 debug shell 一樣的 light-mode wash |

<Aside type="tip" title="Light mode should stay calm">

light mode 的視覺重點應來自層次、邊框、間距與狀態，而不是大面積帶色底。
若需要區分狀態，優先使用局部 accent、badge、icon chip 與 card hierarchy，而不是整頁偏色。

</Aside>

## Starlight Docs Palettes

Astro + Starlight docs may use a separate docs palette layer in addition to Starlight `light` / `dark` mode. The palette switch must map into semantic tokens first, then into Starlight variables.

| Palette | Intent |
| --- | --- |
| `research-mint` | current mint/teal docs UI; calm technical docs default |
| `tech-slate` | high-contrast technical reports and solver diagnostics |
| `nordic-calm` | low-fatigue long-form reports and review material |
| `coffee-terminal` | warm paper / research-log style docs and exploratory reports |

Required docs palette tokens:

- `background`
- `surface`
- `text`
- `muted`
- `accent`
- `rule`

Do not scatter palette hex values across components. Palette values should be centralized in the Starlight stylesheet and exposed to components through semantic CSS variables.

## Agent Rule
```markdown
## Theming
- Use `next-themes` for theme management.
- Provide `light`, `dark`, and `system` modes.
- Prefer semantic tokens such as `background`, `foreground`, `card`, `muted`, `border`, and `primary`.
- Do not hardcode product colors with raw utility choices like `bg-white`, `text-black`, or literal hex values unless there is a documented exception.
- Every component must remain readable in both light and dark themes.
- Chart styling must follow the active theme.
- Theme switching must not trigger avoidable state loss.
- Light mode should prefer a clean white base or warmer neutral base over a cool blue-gray wash.
- Preserve restrained accent fills and icon accents, but do not tint the whole dashboard as if it were a diagnostics shell.
- Starlight docs palettes are `research-mint`, `tech-slate`, `nordic-calm`, and `coffee-terminal`; keep `research-mint` available as the current mint/teal baseline.
- Docs palette switching must be separate from light/dark/system mode, persisted without resetting page state, and implemented through semantic tokens before mapping to Starlight variables.
```
