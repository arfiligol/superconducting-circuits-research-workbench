# Starlight Native Style

Use this page when you create or revise technical docs for the Astro + Starlight zh-TW Build. Astro + Starlight renders the `/docs/` site, while `docs/` remains the editable Source-of-Truth tree.

<Aside type="note" title="How to use this page">

這頁補充 `Standards`、`Information Layout` 與 `Style`。若它們和本頁衝突，以更具體的 owner page 為準；若只是語氣或呈現方式不同，優先採用本頁的 Starlight-native 寫法。

</Aside>

## Page Rhythm

Start each page with a short purpose paragraph. It should say what the topic does and why the reader needs it.

Lead with the happy path, then document caveats, unsupported cases, migration notes, and validation signals. Do not open a current SoT page with removed tools or historical implementations.

Use compact sections. Each section should answer one primary question, and examples should appear close to the workflow they explain.

## Writing Rules

- Use direct, practical wording.
- Lead with the current active contract.
- Keep historical context in notes, warnings, migration sections, or archive pages.
- Include exact commands only after confirming them from repo docs or config.
- Prefer examples over abstract explanation.
- Document failure modes and expected validation signals when they prevent repeated debugging.
- Avoid marketing language, internal team narration, and dense policy prose.

## Starlight Patterns

Use standard Markdown and the repo’s configured Starlight features:

- frontmatter for stable page metadata
- official Starlight `Aside` components for important side information
- official Starlight `Tabs` / `TabItem` components for variants that readers must compare
- official `CardGrid`, `LinkCard`, `Card`, `Steps`, and `FileTree` components when they improve scanning or workflow structure
- native `<details>` for optional long source, long tables, or advanced notes
- Mermaid fences for architecture or workflow diagrams
- overview/index pages with compact cards or tables
- relative Markdown links for internal docs
- fenced code blocks with language identifiers

## Sidebar And Routes

The Starlight sidebar is curated at the top and second level. Do not broad-autogenerate first-level groups.

Use manual groups for `Start Here`, `Workflows`, `Concepts`, `Product App`, `Reference`, and `Contribute & Govern`. Within those groups, use selective `autogenerate` for leaf directories such as `docs/workflows/pluto/`, `docs/app/frontend/`, `docs/concepts/physics/`, or `docs/reference/julia-core/`.

Every editable docs page must be reachable from the Starlight sidebar. A page is covered only when its route appears as an explicit `slug` or lives under an `autogenerate.directory` in `site/astro.config.mjs`. Do not create hidden pages that can only be reached by in-page links or direct URL input.

`Start Here` is notebook-first and should lead with Pluto research setup and the first Pluto notebook. `Product App` is a top-level lane for app contracts, Python Notebook app-prototyping boundaries, frontend, backend, and shared app semantics.

`Start Here`, `Workflows`, and `Concepts` are strict research-core lanes. Do not put App implementation procedures, Backend publication instructions, Schema Editor workflows, Product Async flow, ResultView behavior, TraceStore operations, or WebUI-specific decisions in those lanes. Move that material to `docs/app/**`; keep old App-specific pages under `docs/app/archive/**` with a clear legacy-reference notice.

Every autogenerated child page should provide source frontmatter:

```yaml
sidebar:
  label: Human Friendly Label
  order: 10
```

Avoid raw filename labels such as `fit-model`, `manage-db`, `app-circuit-simulation`, or `schematic-netlist-simulation`. Landing pages should be page maps with reader choices, not copies of the sidebar tree.

MDX is required for pages that use Starlight components such as `CardGrid`, `LinkCard`, `Card`, `Aside`, `Tabs`, `TabItem`, `FileTree`, or `Steps`. Ordinary content pages should stay Markdown unless they need component imports.

<Aside type="caution" title="Do not invent theme syntax">

Do not add per-fence copy syntax, custom table-copy JavaScript, `mkdocs.yml`, or ad hoc Starlight configuration unless the repo has explicitly adopted it.

</Aside>

## Staging Boundary

Edit `docs/` only. The build prepares Starlight content under `site/src/content/docs/docs/`; that staging tree is generated and ignored.

Editable docs source uses official Starlight MDX component syntax directly. The staging transformer keeps frontmatter normalization, first-heading cleanup, and Markdown/MDX link rewriting; it must not be the primary authoring surface for legacy visual syntax.

Legacy MkDocs-style admonitions, collapsible admonitions, tab blocks, and card grids are forbidden in editable docs source. Use `<Aside>` for semantic notes, tips, cautions, and dangers; use native `<details>` only for optional collapsible material.

## Validation

For docs changes, run the docs checks listed in `Build Commands` and `Testing`.
If the change touches navigation, frontmatter, links, or a new docs page, also check that the page appears through the intended directory route.

## Agent Rule
```markdown
## Starlight Native Style
- Write docs under `docs/`; Astro + Starlight renders them at `/docs/`.
- Start each page with a short purpose paragraph and lead with the current active contract.
- Put removed, superseded, or historical implementations in migration notes, warnings, unsupported-case sections, or archive pages, not in the opening narrative.
- Use compact sections, direct practical wording, and examples close to the workflow they explain.
- Use official Starlight component patterns: frontmatter, MDX `Aside`, `Tabs` / `TabItem`, `CardGrid`, `LinkCard`, `Card`, `Steps`, `FileTree`, Mermaid fences, overview cards/tables, relative Markdown links, and fenced code blocks with language identifiers.
- Use native `<details>` only for optional collapsible material such as long source code, long tables, or advanced notes; do not use it for semantic warnings.
- Do not use MkDocs-style admonitions, collapsible admonitions, tab blocks, or card grids in editable `docs/` source.
- Do not edit generated Starlight staging under `site/src/content/docs/docs/`.
- Keep the sidebar manual at top/two-level groups; use `autogenerate` only under curated groups.
- Every editable docs page must be reachable from the sidebar through an explicit `slug` or an `autogenerate.directory`; hidden pages reachable only by URL or in-page links are forbidden.
- Use `Product App` as a top-level group for `docs/app/**`; do not keep active app contracts under `docs/reference/app/**`.
- Keep `Start Here` notebook-first and Pluto-led; do not make Product App setup the primary onboarding path.
- Keep `Workflows` and `Concepts` research-core: physics, Pluto, Julia Core, Python Analysis Core, Python Notebook, research data/evidence, reusable authoring model, and research stack boundaries.
- Move Product App, Backend, Frontend, Schema Editor, ResultView, TraceStore, FastAPI, WebUI, Product Async, and old App workflow procedures to `docs/app/**`; archive superseded App-specific tutorials under `docs/app/archive/**`.
- Add `sidebar.label` and `sidebar.order` to autogenerated children so labels are human-readable and order is intentional.
- Use MDX when Starlight components materially improve a landing/page-map, workflow, reference, or optional-detail experience.
- Do not invent unvalidated theme syntax, per-fence copy hacks, table-copy JavaScript, or site config changes.
- Confirm exact commands from repo docs/config before documenting them.
- Run the relevant docs validation commands after documentation changes.
```
