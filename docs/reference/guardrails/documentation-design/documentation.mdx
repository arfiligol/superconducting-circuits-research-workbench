# Documentation Design

File design specification index to ensure file consistency and maintainability (aligned with the `Astro + Starlight English Docs Build` architecture of this project).

<Aside type="note" title="How to read this section">

Let’s first look at `Standards` and `Macro/Micro Style` to establish common syntax, and then look at `Maintenance` and `Domain Extensions`. This page is the entrance to documentation design, not rewriting every sub-specification.

</Aside>

## Hierarchical division of labor

| Norms | What you are responsible for | What you are not responsible for |
|------|----------|------------|
| Standards | File classification, metadata, core constraints | Page layout, tone details |
| Starlight Native Style | Current contract priority, happy path, examples, Starlight-native rendering rhythm | Diataxis boundaries, build maintenance process |
| API Reference |generated API docs ownership, docstring style, Sphinx/Documenter build boundary| high-level architecture narrative |
| Macro Style / Information Layout | Macro page layout, overview/index rules, reading rhythm | word selection, component syntax details |
| Micro Style / Writing & Visual Elements | Tone, paragraph writing, use of Starlight components / Mermaid and other elements | Diataxis boundaries, build maintenance process |
| Maintenance | Monolingual source tree, frontmatter update, build/check process | Structural design of the file content itself |
| Page Reference Specs | App frontend page The fixed skeleton and acceptance method of specifications | General arrangement of other types of documents |

<Aside type="tip" title="Read order">

If you just want to write a page correctly quickly, the usual sequence is `Standards -> Starlight Native Style -> Macro Style -> Micro Style -> domain-specific extension page`.

</Aside>

## Formal terminology

The site architecture of this project at the file level is officially named:

- `Astro + Starlight English Docs Build`

This term stands for:

- Render `/docs/` using Astro + Starlight
- public artifact built by `site/`, exported to `site/dist/`
- The editing source is `docs/`, and ignored staging tree is generated before building: `site/src/content/docs/docs/`
- generated API reference deployed to sibling sites `/api/python/` and `/api/julia/`
- Editable docs source is English-only; generated/staging output is not an editing source.
