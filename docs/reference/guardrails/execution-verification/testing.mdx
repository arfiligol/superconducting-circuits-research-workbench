# Testing

This page defines the test entry and minimum test expectations of the current platform.

<Aside type="note" title="How to read this page">

First determine whether you are touching `root orchestration`, `app/backend`, `app/frontend`, `Julia Runner`, `desktop` or `docs`.
This page answers "What is the minimum to run", not the complete test design itself.

</Aside>

## Test Map

| Area | Minimum baseline |
| --- | --- |
| Foundation workspace | backend pytest + frontend unit + desktop lint + runner tests |
| Public site | Astro check/build + combined public artifact build |
| Backend | `uv run --package superconducting-circuits-backend pytest app/backend/tests -q` |
| Python analysis | `uv run --package superconducting-circuits-analysis pytest tests/core/analysis -q` |
| Julia Core / Visualizer / Runner | `Pkg.test()` per Julia package |
| Frontend | unit tests, E2E when necessary |
| Desktop foundation | lint + build |
| Docs | source check + build + built-route check |

## Foundation Workspace Check

```bash
uv run --package superconducting-circuits-backend pytest app/backend/tests -q
uv run --package superconducting-circuits-analysis pytest tests/core/analysis -q
npm run test --prefix app/frontend
npm run lint --prefix app/desktop
julia --project=core/julia/SuperconductingCircuitsVisualizer -e 'using Pkg; Pkg.test()'
julia --project=core/julia/SuperconductingCircuitsRunner -e 'using Pkg; Pkg.test()'
JULIA_PYTHONCALL_EXE="$PWD/.venv/bin/python" julia --project=core/julia/SuperconductingCircuitsAnalysisBridge -e 'using Pkg; Pkg.test()'
```

## Backend

```bash
uv run --package superconducting-circuits-backend pytest app/backend/tests -q
```

Backend tests for Runner publication must cover create task, claim task, heartbeat/progress, complete with small staging Zarr manifest, canonical TraceStore publication, TraceRecord creation, trace slice read, invalid manifest rejection, and path traversal rejection.

## Julia Core / Visualizer / Runner

```bash
julia --project=core/julia/SuperconductingCircuitsCore -e 'using Pkg; Pkg.test()'
julia --project=core/julia/SuperconductingCircuitsVisualizer -e 'using Pkg; Pkg.test()'
julia --project=core/julia/SuperconductingCircuitsRunner -e 'using Pkg; Pkg.test()'
```

Runner tests must cover JSON task contract parsing, manifest generation, local Zarr write with small real/imag fixture data, manifest path/shape validation helpers, unsupported task failure, and real task dispatch boundaries.

## Frontend

<Tabs>
<TabItem label="Unit">


```bash
npm run test --prefix app/frontend
```

</TabItem>

<TabItem label="E2E">


```bash
npm run test:e2e --prefix app/frontend
```

</TabItem>

</Tabs>

The foundation workflow currently only requires deterministic unit tests.
Don't use placeholder E2E to pretend to cover the real workflow that hasn't been migrated yet.

<Aside type="caution" title="Frontend review needs real browser evidence">

If the delivered content changes the shell, layout, header/sidebar behavior, dialog/drawer interaction, auth entry, or other obvious user-visible workflow,
Review must not only look at code diff or unit test.
You must use Playwright to walk through the actual process at least once and check whether the UI runs away through screenshots or equivalent visual evidence.

</Aside>

## Public Site

```bash
npm run check --prefix site
npm run build --prefix site
./scripts/build/build_public_site.sh
```

Public site review should verify `/`, `/about/`, and `/docs/` in the combined artifact when the route contract or visible layout changes.

## Desktop Foundation

```bash
npm run lint --prefix app/desktop

```

## Docs

<Aside type="caution" title="Docs changes must always verify routes">

It's not enough for the build to be successful.
Docs changes must pass both source-side nav checking and built-route checking.

</Aside>

```bash
uv run python scripts/check_docs_nav_routes.py --check-source
npm run check --prefix site
npm run build --prefix site
./scripts/build_docs_sites.sh
uv run python scripts/check_docs_nav_routes.py --check-built

# Generated API reference
./scripts/build/build_api_docs.sh
uv run python scripts/build/check_api_reference_links.py
```

## Policy

| Rule | Meaning |
| --- | --- |
| Key workflows must have at least one reproducible test path | Core delivery that relies solely on manual verification will not be accepted |
| Backend service and Runner publication workflow give priority to writing deterministic tests | Ensure task lifecycle and TraceStore publication first |
| The frontend component and interaction process are covered by unit / E2E respectively | Don't use a single test type to shoulder all the responsibilities |
| Frontend merge review requires real UI evidence | User-visible delivery such as shell / layout / overlay / auth entry must be checked at least once with Playwright smoke + screenshot |
| docs route verification must use canonical directory routes | Does not depend on the source `.md` path speculation build results |

<Aside type="tip" title="Good default">

If you are not sure which test to add in this round, first choose the deterministic test closest to the changed surface;
Only add integration / E2E when the user-visible workflow actually spans multiple surfaces.

</Aside>
