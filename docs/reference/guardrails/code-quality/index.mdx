# Code Quality

本區定義 rewrite branch 的程式品質規範。
重點不是追求框架技巧，而是讓 UI、API、Notebook、Julia Runner 與科學核心可以穩定共演化。

<Aside type="note" title="How to use this section">

先用 `Design Patterns` 與 `Contract Versioning` 判斷邊界，再回到 `Code Style`、`Type Checking`、`Error Handling` 等較細的規則。
不要只讀 style 類頁面，就開始改 service boundary 或 contract owner。

</Aside>

## Page Map

| Page | Read this when | Primary concern |
| --- | --- | --- |
| [Code Style](code-style.mdx) | 你在寫或重構一般程式碼 | naming、small functions、clarity |
| [Type Checking](type-checking.mdx) | 你在改 Python / TypeScript 型別 | static guarantees |
| [Design Patterns](design-patterns.mdx) | 你在改 service、repository、DI、layer boundary | architectural consistency |
| [Script Authoring](script-authoring.mdx) | 你在新增或重整 dev/build/test/maintenance helper | scripts structure and responsibility |
| [Data Handling](data-handling.mdx) | 你在動 metadata、trace store、result storage | data boundaries |
| [Logging](logging.mdx) | 你在加 runtime logging | log discipline |
| [Contract Versioning](contract-versioning.mdx) | 你在改 public contract / canonical schema | compatibility strategy |
| [Error Handling](error-handling.mdx) | 你在改 API / worker / CLI error paths | shared error model |

<Aside type="caution" title="Common failure mode">

最常見的錯誤不是「code style 不一致」，而是 UI、API、Notebook、Runner 或 scripts 各自重做同一段 workflow。
只要任務開始碰到 shared workflow，就必須先檢查 `Design Patterns`、`Contract Versioning` 與 `Error Handling`。

</Aside>

## Agent Rule
```markdown
## Code Quality
- 遵循 Clean Code：命名清晰、函式短小、責任單一。
- UI、API、Notebook、Runner、scripts 不得各自複製業務流程；共享規則應集中在 app backend services、Julia Core、Julia Runner 或明確 contract packages。
- 優先修正程式碼而不是增加例外或忽略規則。
- 需要時查閱子文件：Code Style / Type Checking / Design Patterns / Script Authoring / Data Handling / Logging / Contract Versioning / Error Handling。
```
