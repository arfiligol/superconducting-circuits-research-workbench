# Guardrails

This area is the single source of truth for development specifications in the workspace.
The core direction is **Notebook Interface + Electron Application Interface + Julia Runner Compute Plane**.
Python Backend is the control/data plane; Julia Runner is the compute plane; TraceStore is managed by the Backend publication process.

<Aside type="note" title="How to read this section">

Look at `Project Basics` first, then enter `Code Quality`, `UI/UX Quality`, `Execution & Verification` or `Documentation Design` according to the nature of the work.
Don't treat the entire guardrails tree as a one-time must-read list; look for the owner section first, then read the subpages.

</Aside>

<Tabs>
<TabItem label="Human Developers">


- First confirm which concern the current task belongs to.
- Only open sections directly related to the current modification.
- To change the architectural direction, collaboration method, or SoT, first go back to `Project Basics` or `Execution & Verification`.

</TabItem>

## Agent Rule
```markdown
## Guardrails
- `docs/reference/guardrails/` is the workspace rules Source of Truth.
- Use `_agent_catalog.yml` to load only task-relevant guardrails instead of loading the full tree.
- Current architecture is Notebook Interface + Electron Application Interface + Julia Runner Compute Plane.
- Python Backend is the control/data plane; Julia Runner is the compute plane; Backend-managed TraceStore is the official numeric authority.
- Retired command workflow, Python UI runtime, separate queue worker runtime, and Python in-process Julia execution have no authority over current product/runtime surfaces.
- `.agent/rules/` must be synchronized one-to-one from each guardrail file's `## Agent Rule` block whenever guardrail source changes.
```
Installation and loading are two different things; usually only a few necessary rules need to be loaded for a task.
