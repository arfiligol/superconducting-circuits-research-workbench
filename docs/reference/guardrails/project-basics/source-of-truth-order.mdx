# Source of Truth Order

This document defines the decision order of the reference system to prevent humans, Codex or Codex subagents from guessing between shared contract, backend authority, Runner boundary, page spec and implementation.

<Aside type="caution" title="Concern-first resolution">

You cannot decide conflicts by just looking at "which file level is earlier".
First determine which concern this issue belongs to, and then return to the owner file of the concern.

</Aside>

## Canonical Ordering

If descriptions of the same ability conflict with each other, the decision shall be made in the following order:

1. Reference file corresponding to concern owner:
- Product App collaboration / session / auth / workspace / task runtime / audit / common error contract:
   `docs/app/shared/*` + `docs/app/backend/*`
- Product persisted payload / storage schema / field semantics:
   `docs/app/data-contracts/*`
- Public Julia Core authoring invariants:
   `docs/reference/julia-core/*`
- Julia Core / Runner runtime boundary and package invariants:
   `docs/reference/core/*`
- Circuit research route contracts, external-result contract, equivalent-circuit model contract, quantum model boundary, and notebook role boundaries:
   `docs/reference/research-contracts/*`
- User-visible Product App page behavior / page layout / interaction flow:
   `docs/app/frontend/**/*`
- Notebook workflow behavior:
   `docs/reference/notebooks/*`
- Product App frontend/backend/desktop behavior:
   `docs/app/**/*`
2. `docs/app/architecture/contracts/*`’s owner-boundary and canonical contract registry files
3. canonical implementation surface for the concern (for example Runner concern of canonical implementation surface is `core/julia/SuperconductingCircuitsRunner/`)
4. adapters and application implementations:`app/backend/`, `app/frontend/`, `app/desktop/`, `core/`, `notebooks/`
5. retired surfaces and old behavior evidence have no authority over current product contracts

## Scope Boundaries

| Layer | What it owns |
| --- | --- |
| `docs/app/shared/*` |app-level shared semantics, workspace/resource/auth/task runtime/audit/error families|
| `docs/app/backend/*` |app-facing authority surfaces, request/response contract, mutation/read model|
| `docs/app/data-contracts/*` |product persisted record shape, field semantics, storage payload contract|
| `docs/reference/julia-core/*` |Julia Core authoring model, Circuit Plan, components, endpoints, compiler, compiled output, Runner-safe API|
| `docs/reference/core/*` |Julia Core / Runner runtime boundary, installable contract, package-level invariants|
| `docs/reference/research-contracts/*` |external FEM result contract, equivalent-circuit model contract, quantum-model boundary, notebook roles|
| `docs/app/frontend/**/*` |page purpose, layout, interaction, acceptance|
| `docs/reference/notebooks/*` | Notebook research cockpit and inspection workflows |
| `docs/app/architecture/contracts/*` |owner discovery, contract registry, cross-layer boundary map; must not override owner contract|
| implementations | transport, mapping, storage/runtime integration, cannot be reversely rewritten canonical truth |

## Conflict Handling

### Typical Cases

| Conflict situations | Judgment methods |
| --- | --- |
| frontend page spec conflicts with app/shared/backend | `app/shared` + `app/backend` shall prevail; page spec needs to fall back to consumer contract |
| script / old CLI surface conflicts with app/shared | subject to app/shared or backend/runner owner docs; CLI is no longer a product surface |
| The data formats are different from the frontend/backend payload examples | Based on the field semantics of data formats, modify the frontend/backend surface |
| Julia Core / Runner is different from backend/frontend adapter | Repair the adapter first; if the core/runner contract lacks specifications, then fill in the docs and implementation simultaneously |
| External-result or quantum-modeling route conflicts with App data/runtime docs | Use research contracts for research-route boundaries; use App docs only for productized publication/runtime behavior |
| architecture registry is different from owner docs | subject to owner docs, repair registry first |
| product behavior needs an exception | Must be explicitly marked in owner docs and canonical contract registry, not just in the code |
| Retired behavior conflicts with owner SoT | Owner SoT shall prevail; retired behavior does not constitute a product contract |

## Interpretation Rules

- **Owner-first, not consumer-first**:
Page spec, notebook docs, scripts and architecture registry are all important consumers, but they cannot overwrite the owner contract.
- **Reference-first**:
If the reference file conflicts with the implementation behavior, the reference file takes precedence by default.
- **Implementation is not silent law**:
The code path, adapter behavior, and past output are not automatically canonical truth.
- **Julia Core / Runner beats adapters**:
If the Julia Core/Runner contract conflicts with the behavior of the adapter, fix the adapter first; only fix the core/runner and its files if the contract itself is incomplete.
- **Current app/core/notebooks boundaries beat old root layout**:
`app/backend/`, `app/frontend/`, `app/desktop/`, `core/julia/`, `core/python/analysis/`, `core/python/circuit_libraries/`, and `notebooks/` are architecture-level surfaces.
If root `backend/`, `frontend/`, `desktop/`, `cli/` or `src/` still has content, it can only be regarded as retired residue and canonical topology cannot be deduced in reverse.
- **Do not silently revise docs to match code**:
When it is found that the implementation and reference are inconsistent, the file cannot be directly modified to match the code, unless the user explicitly confirms that the specifications need to be changed.
- **Exceptions must be explicit**:
If it is determined that a product contract requires a special exception, it must be explicitly recorded in the owner docs and canonical contract registry, and cannot just be left in the program code.
- **Legacy fallback is not a product contract**:
Unless explicitly required by the owner SoT, the old behavior, old data, or retired adapter reverse requirement current product path shall not be used to retain the compatibility layer.

## Agent Rule
```markdown
## Source of Truth Order
- Resolve conflicts by concern owner first:
  - app collaboration/session/auth/workspace/task runtime/audit/error -> `docs/app/shared/*` + `docs/app/backend/*`
  - product persisted payload/schema fields -> `docs/app/data-contracts/*`
  - Julia Core authoring invariants -> `docs/reference/julia-core/*`
  - Julia Core / Runner runtime boundary and package invariants -> `docs/reference/core/*`
  - circuit research route contracts, external-result contract, equivalent-circuit model contract, quantum-model boundary, notebook role boundaries -> `docs/reference/research-contracts/*`
  - page behavior/layout -> `docs/app/frontend/**/*`
  - notebook workflow behavior -> `docs/reference/notebooks/*`
- Use `docs/app/architecture/contracts/*` only as owner-boundary and canonical contract registry guidance, not as the primary owner when owner docs already exist.
- Treat implementation and old behavior as evidence, not automatic canonical truth.
- If owner docs and consumer docs conflict, prefer the owner docs unless the user explicitly changes the spec.
- If Julia Core / Runner and adapters conflict, fix the adapter first unless the canonical contract is incomplete.
- Treat root-level `backend/`, `frontend/`, `desktop/`, `cli/`, and `src/` residues as retired surfaces with no authority over product boundaries; current architecture-level surfaces include `core/python/analysis/` and `core/python/circuit_libraries/`.
- Record intentional product-contract exceptions in the owner docs and canonical contract registry.
- Legacy fallback is not a product contract unless an owner SoT explicitly requires it.
```
