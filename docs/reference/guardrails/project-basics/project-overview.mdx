# Project Overview

The product architecture of this project is **Notebook Interface + Electron Application Interface + Julia Runner Compute Plane**.
Legacy command workflow, retired Python UI runtime, separate queue worker runtime and Python-in-process Julia execution are not included in the product placement.

<Aside type="note" title="How to read this page">

First use this page to confirm the product mission and core surface, and then look at the execution layer guardrails such as `Tech Stack`, `Folder Structure`, and `Backend Architecture`. This page sets product boundaries, not implementation details.

</Aside>

## Mission

Create a superconducting circuit platform that allows researchers to accomplish the following tasks with clear layering:

- Data Browser
- Dataset / Trace / Result Browser
- Task / Execution Center
- Simulation Workbench
- Analysis Workbench
- Notebook research cockpit
- Julia simulation and analysis runner
- TraceStore-backed numeric result management

## Product Goals

The core goal of this product is not to simply rewrite the UI, but to establish a maintainable, scalable, and traceable superconducting circuit research work platform.

- Python Backend is control plane + data plane, responsible for task lifecycle, metadata, publication, provenance, TraceStore registration and data API
- Julia Runner is compute plane, owns simulation, parameter sweep, post-processing, analysis, fitting, derived parameter extraction and result package generation
- Electron Application is productized data, simulation, and analysis workbench, focuses on Simulation Workbench, Analysis Workbench, dataset, ingestion, trace browsing, Task / Execution Center and ResultView
- Pluto Notebook is a direct Julia research cockpit that can directly execute Julia Core
- Python Notebook is Product App prototyping and inspection surface. It can directly read data for temporary analysis, and can also verify Backend/task/ResultView assumptions; if you want to change the platform status, you must use Backend contract
- user-facing command workflow is no longer a product surface; the repo only retains `scripts/` as dev/build/test/maintenance helpers

## Interface Workflow Goals

The project must support the following research workflows, rather than just provide piecemeal tools:

- Use Julia Core directly in Pluto for research-based simulation and analysis
- Submit persisted simulation tasks from Application Simulation Workbench
- Submit persisted analysis / fitting / post-processing tasks from Application Analysis Workbench
- Inspect local/exported/canonical data files directly from the Python Notebook and call the Backend data/task/result APIs when platform semantics or state modification is required
- Manage dataset / design / trace / task / result / provenance
- Let Runner staging result be published into a formal TraceStore batch after being verified by Python Backend
- Allow results to be saved, traced, re-attached, re-analyzed and compared

## Data And Provenance Goals

The data plane goal of this project is to clearly separate the relationship between metadata, trace payload and results, while maintaining rebuildability.

- metadata is managed by the database
- Numeric payload and trace are managed by TraceStore Zarr managed by Python Backend
- Runner only writes the local filesystem staging Zarr package and does not write the formal metadata DB
- Any simulation / post-process / analysis results must be traceable to dataset, design, trace, task, batch and provenance
- frontend must not be the sole holder of canonical computation state
- HTTP/JSON must not transmit large ND arrays; only task control, status, manifest locator, summary and slice/detail API payloads

## System Success Criteria

When the overall reconstruction is completed, at least the following must be established at the same time:

- legacy command workflow, retired Python UI runtime, separate queue worker runtime, and Python in-process Julia runtime are no longer active product/runtime surfaces
- backend can independently provide auth/session/workspace, dataset/design/trace metadata, tasks, TraceStore publication and frontend/notebook data API contracts
- Julia Core becomes reusable circuit construction / simulation / analysis library
- Runner fixture outputs are test utilities only, not product task kinds; unsupported product task kinds fail loudly
- frontend only retains draft state / interaction state / view state, but does not save canonical computation state
- Electron only functions as a desktop shell, local mode starts frontend, Python Backend and Julia Runner, and does not start separate queue service
- task/dataset/result can be reattached and rebuilt after refresh, reconnect, and restart.

<Aside type="tip" title="Success bar">

The completion condition for rewrite is not "change the screen to Next.js", but that Python Backend publication, Julia Runner staging, TraceStore provenance, Notebook research cockpit and Electron data workbench can all be established under the same set of boundaries.

</Aside>

## Scope

### Core Product Surfaces

| Capabilities | Description |
| --- | --- |
| Data Browser | Browse metadata, trace summary, analysis result and lineage |
| Dataset | Manage dataset, design, metadata and provenance |
| Simulation Workbench | Submit productized simulation request, monitor task, view published result |
| Data Ingestion |Import measurement/layout/simulation artifacts|
| Raw Data / Trace Browser |Browse TraceStore metadata, summary, slice and result lineage|
| Task / Execution Center |Monitor async Runner task, execution history, Runner runtime summary and publication result handoff|
| Analysis Workbench | Create analysis / fitting / post-processing request, submit persisted task, and present the results through ResultView |
| Notebook Interface |Pluto direct research cockpit; Python notebooks for Product App prototyping, data/file inspection, Backend API inspection, migration, emergency analysis|

### Accepted Data Sources

- circuit simulation traces
- layout simulation traces (e.g. HFSS/Q3D)
- measurement traces (e.g. VNA)
- Compatible S/Y/Z matrix traces and their derived analysis results

### Rewrite Direction

- UI: Mainly based on `app/frontend/`’s **Next.js App Router**
- API: Mainly **FastAPI** of `app/backend/`
- Compute: Use `core/julia/SuperconductingCircuitsRunner/` as async compute plane
- Core: `core/julia/SuperconductingCircuitsCore/` as reusable Julia library
- Desktop: wrap frontend, backend, runner local mode in **Electron** of `app/desktop/`
- Notebook: Use `notebooks/pluto/` and `notebooks/python/` as research and inspection cockpit
- Retired surfaces: existing command workflow, retired Python UI runtime, separate queue worker runtime, Python in-process Julia execution active entrypoint

<details class="docs-disclosure docs-disclosure--note">
<summary>Command workflow position</summary>

User-facing command workflow has been removed from product surface. Actions that require automation should be placed in `scripts/dev/`, `scripts/build/`, `scripts/test/` or `scripts/maintenance/`, and must not be packaged into user workflow contracts.

</details>

## Target Audience

- Superconducting circuits and quantum hardware researchers
- Users who need to integrate simulation / layout / measurement
- Developers who need reproducible notebook workflows, async runner pipelines, and extensible web UIs
