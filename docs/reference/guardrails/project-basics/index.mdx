# Project Basics

This area defines the basic consensus of the platform: product scope, technology selection and repo structure.
Any changes that affect the overall development direction should be preceded by updating these files.

<Aside type="note" title="What this section owns">

`Project Basics` defines the common premise of the entire workspace.
If the question is asking "What exactly are we doing, what are we using, where should the files be placed, and who has the final say in case of conflicts?", the answer should be found here first.

</Aside>

## Page Map

| Page | Read this when | Core question |
| --- | --- | --- |
| [Project Overview](project-overview.mdx) | You need to confirm product boundaries, main workflows, and rewriting direction | What exactly is this platform going to do? |
| [Tech Stack](tech-stack.mdx) | Do you want to confirm the framework, dependency or runtime baseline | What technologies do we officially adopt? |
| [Folder Structure](folder-structure.mdx) | You want to add files, move modules, and reorganize directories | Where should these changes be made? |
| [Backend Architecture](backend-architecture.mdx) | You want to change the API / services / persistence / facade boundary | How should the backend be layered? |
| [Source of Truth Order](source-of-truth-order.mdx) | You encounter a conflict in docs, backend, frontend, Runner, Notebook | Who has the final say in a conflict? |

<Aside type="tip" title="Read order">

Let’s look at `Project Overview` first, then `Tech Stack` and `Folder Structure`.
If the task directly encounters a contract conflict or owner boundary, look at `Source of Truth Order` and `Backend Architecture`.

</Aside>

## Agent Rule
```markdown
## Project Basics
- Project Basics defines the mission, scope, technology stack and structure of the platform.
- Any changes that affect overall collaboration and architectural consistency must first update this area.
- The current public introduction site direction is Astro; the product app UI direction is Next.js, the API direction is FastAPI, the compute plane direction is Julia Runner, and the Notebook is a research cockpit.
- The backend's responsibility boundaries and internal blueprint are defined by `backend-architecture.md`.
- The old command workflow, retired Python UI runtime, separate queue worker runtime and Python in-process Julia runtime are not part of the product contract and should no longer be the default landing point for new features.
```
