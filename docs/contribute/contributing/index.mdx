# How to contribute (Contributing)

Thank you for your interest in participating in the development of the **Superconducting Quantum Circuit Platform** project.

This page only provides contributor-facing quick entry:

- How to set up a development environment
- How to perform basic checks
- How normal contributions should be aligned branch / worktree flow
- How to use Codex for docs-first, multi-agent, broad-slice + fixup iteration

complete branch roles, isolated worktree policy, merge authority and multi-agent delivery model,
Please refer to the following two pages:

- [Branch & Worktree Flow](../../reference/guardrails/execution-verification/branch-and-worktree-flow.mdx)
- [Codex Subagent Coordination](../../reference/guardrails/execution-verification/multi-agent-collaboration.mdx)

If you want to directly use the Codex collaboration workflow common in this project, please refer to:

- [Division of labor and Fixup workflow using Codex](codex-agent-workflow.mdx)

## 1. Environment setup

We use `uv` for dependency management.

### Prerequisites
- Python 3.12+
- [uv](https://github.com/astral-sh/uv)

### Installation steps

1. **Fork Project**:
Click "Fork" on the GitHub project page to copy the project to your own account.

2. **Clone your Fork**:
Replace `<YOUR_USERNAME>` with your GitHub account:
  ```bash
  git clone https://github.com/<YOUR_USERNAME>/superconducting-circuits-research-workbench.git
  cd superconducting-circuits-research-workbench
  ```

3. **Sync dependencies**:
This will automatically set up the `.venv` virtual environment and install all packages (including development tools).
  ```bash
  uv sync --all-packages
  ```

4. **Start the virtual environment**:
  ```bash
  source .venv/bin/activate
  ```

## 2. Development process (Workflow)

### Branch / Worktree Quick Summary

The short version rules of general contributor flow are as follows:

1. First synchronize the latest status from `develop`.
2. The new work defaults to creating feature branch from `develop`.
3. If AI agents are used, **each agent/task must use an independent worktree**.
4. The default PR target for normal implementation / docs / test work is `develop`.
5. `main` is a release branch, not the default landing branch for daily feature work.

<Aside type="note" title="Canonical policy lives in Reference">

This page does not repeat the full Git/worktree policy.
If you want to confirm the role of `main` / `develop`, the agent merge authority, or the bounded write root of the sandbox/example agent,
Please read directly [Branch & Worktree Flow](../../reference/guardrails/execution-verification/branch-and-worktree-flow.mdx).
It also clearly stipulates that: the source code in `sandbox/**` / `examples/**` is traceable content; the disposable outputs such as `csv`, `svg`, `png` generated by it are not the default must-track deliverables.

</Aside>

### Execute test
We use `pytest` to ensure proper functionality.
```bash
uv run pytest
```

### Preview file
Technical documentation created by Astro + Starlight, source still at `docs/`. You can start Starlight dev server directly during development:

```bash
npm run dev --prefix site
```

Formal static output will be generated at `site/dist/docs/` and passed route check:

```bash
./scripts/build_docs_sites.sh
```

To preview the same combined artifact as GitHub Pages, that is, `/`, `/docs/`, `/api/python/`, `/api/julia/` all in the same output:

```bash
uv run docs-site serve --build
```

Starlight dev server supports Hot Reload, and the page will be automatically refreshed after saving the file.

### Submit changes (Pull Request)

1. **Create Branch**:
  ```bash
  git fetch origin
  git switch develop
  git pull --ff-only origin develop
  git switch -c feature/my-new-feature
  ```
2. **Commit changes (Commit)**:
We recommend following the Conventional Commits specification (e.g., `feat:`, `fix:`, `docs:`).
  ```bash
git commit -m "feat: add a new feature"
  ```
3. **Push to Fork (Push)**:
  ```bash
  git push origin feature/my-new-feature
  ```
4. **Initiate a Pull Request**:
Return to the original project page and click "Compare & pull request" to submit your changes.
The default PR target for normal work should be `develop`; `main` is reserved for verified release promotion.

### Pre-commit Checks
Before submitting a Pull Request, please make sure:

1. **Typing**: Pass type checking (we use Python 3.12+ syntax).
2. **Formatting**: The code has been formatted (following PEP 8, checked by Ruff).
3. **Tests**: All tests passed.

> For detailed code quality standards, please refer to [Code Style Guardrails](../../reference/guardrails/code-quality/code-style.mdx).
