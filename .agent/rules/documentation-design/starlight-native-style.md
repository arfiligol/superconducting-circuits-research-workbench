## Starlight Native Style
- Write docs under `docs/`; Astro + Starlight renders them at `/docs/`.
- Keep editable docs source English-only; generated/staging output is not an editing source.
- Start each page with a short purpose paragraph and lead with the current active contract.
- Put removed, superseded, or historical implementations in migration notes, warnings, unsupported-case sections, or archive pages, not in the opening narrative.
- Use compact sections, direct practical wording, and examples close to the workflow they explain.
- Use official Starlight component patterns: frontmatter, MDX `Aside`, `Tabs` / `TabItem`, `CardGrid`, `LinkCard`, `Card`, `Steps`, `FileTree`, Mermaid fences, overview cards/tables, relative Markdown links, and fenced code blocks with language identifiers.
- Use native `<details>` only for optional collapsible material such as long source code, long tables, or advanced notes; do not use it for semantic warnings.
- Do not use MkDocs-style admonitions, collapsible admonitions, tab blocks, or card grids in editable `docs/` source.
- Do not edit generated Starlight staging under `site/src/content/docs/docs/`.
- Keep the sidebar manual at top/two-level groups; use `autogenerate` only under curated groups.
- Every editable docs page must be reachable from the sidebar through an explicit `slug` or an `autogenerate.directory`; hidden pages reachable only by URL or in-page links are forbidden.
- Use `Product App` as a top-level group for `docs/app/**`; do not keep active app contracts under `docs/reference/app/**`.
- Keep `Start Here` as a two-step student/research onboarding path: first Pluto notebook, then reusable circuit design. Do not make Product App setup the primary onboarding path.
- Keep `Workflows` and `Concepts` research-core: physics, Pluto, Julia Core, Python Analysis Core, Python Notebook, research data/evidence, reusable authoring model, and research stack boundaries.
- Keep non-app `Reference` research-core: API landing, core packages, Julia Core, Julia Visualizer, notebook research interface, utilities, agent skills, contributors, and guardrails. Product architecture/data/search/runtime/UI contracts belong under `docs/app/**`.
- Move Product App, Backend, Frontend, Desktop, Electron, Schema Editor, ResultView, TraceStore, FastAPI, WebUI, Product Async, Data Search/Data Management, and old App workflow procedures to `docs/app/**`; archive superseded App-specific tutorials under `docs/app/archive/**`.
- Add `sidebar.label` and `sidebar.order` to autogenerated children so labels are human-readable and order is intentional.
- Use MDX when Starlight components materially improve a landing/page-map, workflow, reference, or optional-detail experience.
- Do not invent unvalidated theme syntax, per-fence copy hacks, table-copy JavaScript, or site config changes.
- Confirm exact commands from repo docs/config before documenting them.
- Run the relevant docs validation commands after documentation changes.
